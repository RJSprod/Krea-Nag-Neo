TECHNICAL DEVELOPMENT SPEC
Forge Neo extension: NAG support for Krea2 Turbo / Krea2 Raw
Date: 2026-07-02
Prepared for: coding agent implementation

================================================================================
0. Purpose
================================================================================
Build a Forge Neo WebUI extension that adds NAG, meaning Normalized Attention
Guidance, for Krea2-family image generation.

The extension must prioritize Krea2 Turbo because Krea2 Turbo is an 8-step
distilled checkpoint whose official inference examples run with CFG disabled.
That makes normal negative prompting weak or unavailable, and it is the exact
few-step/distilled use case NAG was designed to address.

This document intentionally scopes only the first three requested priorities.
Treat all lower-bit/exotic paths and optimized-attention support as future work.

2. Requested support scope
================================================================================
Required priorities:

P0. Foundation / base MVP
    - Extension installs and loads in Forge Neo.
    - UI appears in txt2img for Krea2-family models.
    - No-op behavior is stable when disabled or when the active model is not Krea2.

P1. Krea2 Turbo bf16
    - Must work with official Turbo-style settings:
        steps: 8
        CFG/guidance: 0.0 or disabled
        mu: 1.15 when the UI exposes mu
        resolution: 1024 to 2048, starting with 1024
    - This is the main target.

P2. Krea2 Turbo fp8_scaled
    - Must work with Forge Neo's fp8_scaled Krea2 Turbo loading path.
    - NAG math should operate on attention activations in a safe dtype, preferably
      float32 for norms and then cast back to the model activation dtype.
    - Do not require nvfp4, mxfp8, int8, convrot int8, or GGUF in this milestone.

P3. Krea2 Raw bf16 + LoRA compatibility
    - Raw bf16 support should not regress Turbo.
    - LoRA compatibility means: when Forge Neo has already applied/loaded a Krea2
      LoRA, enabling NAG must not strip or bypass the LoRA-modified weights.
    - Required LoRA path: Raw-trained LoRAs used on Turbo inference, matching Krea's
      recommendation. Also verify Raw bf16 with LoRA does not crash.

Explicit non-goals for this spec:
    - Krea2 Turbo nvfp4, mxfp8, int8, convrot int8.
    - Wan, Flux, SDXL, SD3, Qwen-Image, Anima, Z-Image, Ernie-Image.
    - Optimized-attention compatibility as a release requirement: SageAttention,
      FlashAttention, Spectrum, TeaCache, WaveSpeed, torch.compile full-model.
    - img2img, inpaint, ControlNet, refiner/high-low routing.
    - Perfect speed parity with normal Turbo sampling.

    krea2_raw
    force_disable
- Debug logging: checkbox, default false.

Infotext metadata must include:
    Krea2 NAG: enabled/disabled
    NAG negative prompt
    nag_scale
    nag_tau
    nag_alpha
    nag_sigma_end or equivalent timestep cutoff
    Krea2 NAG mode detected: turbo/raw/unknown

4. High-level architecture
================================================================================
Do NOT implement this as normal CFG. The extension must apply NAG inside the
attention path.

Preferred architecture:

sd_forge_krea2_nag/
  install.py                         optional; keep dependencies minimal
  scripts/
    krea2_nag.py                     WebUI Script class + UI + process hooks
  krea2_nag/
    __init__.py
    state.py                         request-local NAG state
    detect.py                        active model detection
    conditioning.py                  negative prompt encoding helpers
    patcher.py                       patch/restore attention modules
    attention.py                     Krea2 attention wrapper + NAG math
    logging.py                       debug logs
    version.py
  README.md
  LICENSE                            include MIT attribution if code is adapted

Preferred Forge hook sequence:

1. In Script.ui(), render the UI controls.
2. In process_before_every_sampling or the closest Forge Neo callback:
   - Return immediately if disabled.
   - Detect active model family.
   - Return or warn if not Krea2.
   - Encode the NAG negative prompt with the same Krea2/Qwen3-VL text encoder path
     used by the current model.
   - Install Krea2 attention wrappers on the active model.
   - Store request-local state; do not use persistent global state without cleanup.
3. During model forward:
   - Duplicate the image/latent branch for positive and NAG-negative conditioning.
   - Run attention with both positive and NAG-negative text context available.
   - Apply NAG feature extrapolation/normalization/blending to the positive image
     token features.
   - Return only the guided positive velocity/prediction to the sampler, plus
     normal uncond branch if CFG is active.
4. After sampling, restore all patched functions/modules in a finally block.

Important: patch after Forge Neo has loaded the model and after LoRA patches have
been applied. Restore the exact original functions on completion or error.

================================================================================
6. Core NAG math
================================================================================
Use the official NAG implementation as the source of truth, especially
attention_joint_nag.py.

The essential math, expressed for matching positive and negative attention
features, is:

    z_guided = z_positive * nag_scale - z_negative * (nag_scale - 1)

    norm_positive = L1Norm(z_positive, dim=-1, keepdim=True)
    norm_guided   = L1Norm(z_guided,   dim=-1, keepdim=True)

    ratio = norm_guided / (norm_positive + eps)

    z_normalized = z_guided * min(ratio, nag_tau) / (ratio + eps)

    z_out = z_normalized * nag_alpha + z_positive * (1 - nag_alpha)

Implementation details:

- Compute L1 norms in float32 for fp8_scaled/bf16 stability.
- Cast z_out back to the original attention activation dtype.
- Apply only when nag_scale > 1.0 and a non-empty NAG negative conditioning exists.
- Start by applying guidance to image-token attention features only. For Krea2's
  single-stream text+image sequence, text-token sequence lengths may differ between
  positive and negative prompts, but image-token length is stable.
- Leave text-token outputs unchanged unless experiments prove guiding text tokens is
  necessary and stable.
- Add eps = 1e-7 or similar.

================================================================================
7. Krea2-specific implementation strategy
================================================================================
Krea2 is a SingleStreamDiT / MMDiT-style model. The reference code concatenates
text context tokens and image tokens into a single sequence before transformer
blocks. Therefore, the extension must be Krea2-family specific.

Do not assume a cross-attention module like SDXL UNet. The target is the
single-stream self-attention over combined text+image tokens.

Reference Krea2 attention path from official mmdit.py:

- Attention.forward(qkv, freqs, mask)
- q/k/v/gate are projected from qkv.
- q/k/v are reshaped into heads.
- q/k are normalized.
- RoPE is applied when freqs is provided.
- scaled_dot_product_attention is called.
- attention output is rearranged back to sequence features.
- output is multiplied by sigmoid(gate).
- output projection wo is applied.

Recommended patch point:

Patch the Krea2 Attention.forward method or equivalent Forge Neo Krea2 attention
module. The wrapper should reproduce the original code path, but before output
projection it should apply NAG to the positive image-token feature slice using the
corresponding negative image-token feature slice.

Do not monkeypatch torch.nn.functional.scaled_dot_product_attention globally.
Patch only the Krea2 model instance's attention modules.

Pseudo-flow for Turbo CFG-disabled case:

    positive_context, positive_txt_mask = existing conditioning for prompt
    nag_context, nag_txt_mask = encode(nag_negative_prompt)
    img_tokens = current latent image tokens

    Build a 2-branch Krea2 forward:
        branch 0: positive_context + img_tokens
        branch 1: nag_context      + img_tokens

    For each Krea2 block's attention output:
        z_positive_img = attention_output[branch 0, image_token_slice, :]
        z_negative_img = attention_output[branch 1, image_token_slice, :]
        z_guided_img = nag(z_positive_img, z_negative_img)
        replace branch 0 image-token output with z_guided_img

    Continue block computation.
    At model output:
        return only branch 0's final image-token velocity/prediction.

Pseudo-flow for Raw + CFG case:

    Branches may be:
        branch 0: uncond / normal negative prompt for CFG
        branch 1: positive prompt
        branch 2: NAG negative prompt

    Apply NAG to branch 1 using branch 2.
    Leave branch 0 unchanged.
    Return branch 0 and guided branch 1 to Forge's existing CFG composition.

If Forge Neo uses a different ordering for cond/uncond branches, detect and adapt
rather than assuming this branch order. Add debug logs showing branch count and
mode.

================================================================================
8. Conditioning and text-encoder requirements
================================================================================
The NAG negative prompt must be encoded with the same Krea2 text encoder path as
normal prompts.

Krea2 uses Qwen3-VL conditioning in the official code. In Forge Neo, find the
existing Krea2 text-encoding wrapper and call that, rather than importing a second
copy of Qwen3VLConditioner.

Requirements:

- Do not instantiate a duplicate Qwen3-VL text encoder unless no Forge pathway is
  available. Reusing the active model pipeline is required for memory efficiency.
- Cache NAG negative conditioning per prompt/batch/seed request, not globally.
- Batch behavior:
    - For batch size N, encode N NAG-negative prompts.
    - If the UI field is one prompt, repeat it N times.
    - If prompt matrix/batch prompt behavior is available, initially repeat the
      same NAG negative prompt for all items.
- Text length mismatch handling:
    - If Krea2/Forge produces fixed-length text embeddings, use them directly.
    - If positive and NAG negative contexts differ in token length, pad both to
      the same text length and update masks correctly.
    - Image-token slices must be calculated from the actual text length after
      padding.
- Empty NAG negative prompt:
    - Either no-op or use an empty-string encoded negative prompt based on a UI
      setting.
    - Default should be no-op for empty field to avoid surprising users.

================================================================================
9. Dtype and quantization requirements
================================================================================
P1 bf16:
    - Model weights/activations usually bf16.
    - NAG norm calculations should use float32 internally, then cast to bf16.

P2 fp8_scaled:
    - Treat fp8_scaled as a model weight/loading format. Attention activations may
      be bf16/fp16/float32 depending on Forge Neo internals.
    - Do not assume activations are actually torch.float8 tensors.
    - If an activation tensor is fp8 or unsupported for torch.norm, upcast to
      float32 for NAG math and cast back to the original dtype after blending.
    - Validate no dtype mismatch around output projection.

P3 LoRA:
    - Apply NAG wrappers to the active model object after LoRA has modified or
      injected layers.
    - Do not clone a pristine model that loses LoRA patches.
    - If Forge applies LoRA dynamically per request, install NAG after LoRA
      application or wrap the post-LoRA attention modules.
    - Test with at least one official Krea LoRA if available locally.

================================================================================
10. Extension lifecycle and cleanup
================================================================================
The extension must be safe to enable/disable without restarting Forge Neo.

Required lifecycle behavior:

- Store original attention forward callables in a patch registry.
- Patch only once per module per request.
- Use try/finally cleanup around generation.
- Restore original callables after generation completes or errors.
- Clear request-local NAG state after generation.
- Do not leave model_options, sampler hooks, or global state mutated.
- If another attention-guidance extension is active, warn and skip unless explicit
  override is enabled.

Potential known collision area:
- Forge Neo users have reported conflicts between PAG/SAG-style extensions and
  integrated features such as Spectrum. Because NAG is also an attention-level
  modification, prioritize clean restoration and incompatibility warnings.

- Non-Krea2 model: silently no-op or show one warning in debug mode.

14. Tuning defaults
================================================================================
Start with these defaults; tune empirically.

For Krea2 Turbo:
    nag_scale = 5.0
    nag_tau   = 2.5
    nag_alpha = 0.125 or 0.25
    apply across all 8 steps initially

For Krea2 Raw:
    nag_scale = 3.0 to 5.0
    nag_tau   = 1.5 to 2.5
    nag_alpha = 0.10 to 0.25
    If CFG is also active, start lower than Turbo to avoid overguidance.

Expected UI guidance:
    - Increase nag_scale for stronger suppression.
    - Increase nag_tau or nag_alpha carefully; too high can create artifacts.
    - For few-step Turbo, stronger tau/alpha is usually more tolerable than for
      many-step Raw.

================================================================================
15. Common pitfalls
================================================================================
1. Accidentally implementing CFG instead of NAG.
   - Do not return cond + scale * (cond - neg) at the model-output level and call it
     done. NAG must operate inside attention features.

2. Text length mismatch.
   - Krea2 positive and NAG negative prompts may encode to different lengths.
   - Guide the image-token slice because image-token length is stable.
   - Pad text contexts/masks if batching positive and negative contexts together.

3. Broken cleanup.
   - If the user must restart Forge Neo after one NAG run, cleanup is insufficient.

4. Breaking LoRA.
   - Patching must wrap post-LoRA modules, not replace them with pristine modules.

5. Optimized attention assumptions.
   - Sage/Flash/Spectrum may alter attention shapes or kernels.
   - Out of scope until explicitly tested.

6. Batch layout confusion.
   - Forge samplers may already batch cond/uncond branches.
   - Add debug logging for branch count and tensor shapes.
   - Fail closed instead of silently generating corrupted results.

7. Applying NAG after output projection only.
   - NAG should be applied to attention features before or at the attention-output
     feature stage. Applying to final denoiser output is not the intended method.

    def process_before_every_sampling(self, p, *args):

        if not info.is_krea2:
            return
        nag_cond = encode_krea2_nag_negative(p, model, nag_negative_prompt)
        state = Krea2NagState(...)
        install_krea2_attention_patches(model, state)
        p.extra_generation_params.update({...})

    def postprocess(self, p, processed, *args):
        restore_krea2_attention_patches(p.sd_model)

Use a try/finally pattern where Forge callbacks permit it. If postprocess is not
guaranteed after failure, register an additional cleanup callback or wrap the
sampler launch path the way existing Forge Neo extensions do.

================================================================================
18. Definition of done
================================================================================
The extension is complete for this scope when:

- It installs by copying the folder into Forge Neo's extensions directory and
  restarting Forge Neo.
- It displays a Krea2 NAG UI section.
- It no-ops safely on non-Krea2 models.
- It supports Krea2 Turbo bf16 with CFG disabled at 8 steps.
- It supports Krea2 Turbo fp8_scaled without dtype errors.
- It supports Krea2 Raw bf16.
- It supports Krea2 LoRA compatibility without stripping LoRA behavior.
- It writes reproducible infotext metadata.
- It restores all patches after generation and after errors.
- It has debug logs sufficient to diagnose branch layout, dtype, text/image token
  slices, and whether NAG was applied.
- It includes README warnings about unsupported low-bit modes and optimized
  attention paths.
